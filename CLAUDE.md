## Route Groups

Next.js route groups (`(public)`, `(admin)`, `(child)`) segment routes by role without affecting the URL path.

- `(public)` вЂ” No auth required. Login page lives here.
- `(admin)` вЂ” Requires `role === 'adult'`. Protected by layout middleware.
- `(child)` вЂ” Requires `role === 'child'`. Protected by layout middleware.

---

## Auth Model

- **Library:** iron-session v8 (cookie-based, encrypted, server-side)
- **No JWT:** Sessions are stored in encrypted cookies. No tokens to manage.
- **Password hashing:** bcryptjs with salt rounds = 12
- **Session shape:**

```ts
interface SessionData {
  userId: string
  role: 'adult' | 'child'
  name: string
}
```

- Route protection is enforced in each route group's `layout.tsx` by reading the session via a server component helper.

## Reward System

Three separate reward channels вЂ” all derived from `approved_points`:

| Reward | Formula | Purpose |
|--------|---------|---------|
| Allowance | `points Г· ALLOWANCE_POINTS_PER_DOLLAR Г— 100` cents | Real money, adult-controlled |
| XP | `points Г— XP_PER_POINT` | Permanent character level |
| Scrap (game currency) | `points Г— GAME_CURRENCY_PER_POINT` | In-game purchases only |

Conversion rates are set via environment variables and can be tuned without code changes.

## Chore Lifecycle

```
available в†’ submitted_complete в†’ approved вњ“
                               в†’ rejected вњ—
available в†’ missed (daily job marks overdue)
available в†’ expired (assignment ended)
```

- Only `approved` instances contribute to points/XP/scrap.
- Weekly finalize looks at all `approved` instances for the week.

---

## Weekly Finalize Flow

1. Adult opens weekly review for a child + week
2. System aggregates all `approved` ChoreInstances for that period
3. System adds any `approved` BonusRequests
4. Totals computed: points, allowance, XP, scrap
5. Adult confirms
6. System:
   - Creates `WeeklyLedger` row (unique on `childUserId + weekStart`)
   - Updates `User.allowanceBalanceCents`, `User.xp`, `User.gameCurrencyBalance`
   - Evaluates level-up threshold
   - Evaluates story chapter unlocks
   - Creates `AuditLog` entry
7. Duplicate finalization blocked by the unique constraint

---

## Story Chapter Unlock Rules

Stored as JSON in `StoryChapter.unlockRulesJson`. Example structure:

```json
{
  "minLevel": 2,
  "minApprovedChores": 20,
  "requiredUpgrades": ["nail-boards"]
}
```

Evaluated whenever XP or approved chore counts change (on chore/bonus approval and on weekly finalize).

## API Surface

All mutations use Next.js Route Handlers in `app/api/`. All routes validate input with Zod and check session/role before executing.

Key endpoints (implemented in Phase 3+):
- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`
- `GET /api/child/dashboard`
- `POST /api/child/chore-instances/[id]/submit`
- `POST /api/child/bonus-requests`
- `GET /api/admin/approvals`
- `POST /api/admin/chore-instances/[id]/approve`
- `POST /api/admin/weekly-review/finalize`

## Conventions

- **Server components by default.** Use `'use client'` only when needed (forms, animations, interactive state).
- **Zod validation on every mutation.** Schema lives in `lib/validation/`.
- **Prisma client is a singleton** via `lib/db/index.ts`. Never instantiate `PrismaClient` elsewhere.
- **No hardcoded secrets.** All config comes from environment variables.
- **Audit log on every adult approval action.** Use the `createAuditLog` helper in `lib/auth/`.
